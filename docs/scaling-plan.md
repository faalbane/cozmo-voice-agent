## Auto-Scaling Configuration (Kubernetes HPA)

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: voice-agent-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: voice-agent
  minReplicas: 3
  maxReplicas: 70
  metrics:
    - type: Pods
      pods:
        metric:
          name: voice_agent_concurrent_calls
        target:
          type: AverageValue
          averageValue: "15"
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 65
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30
      policies:
        - type: Pods
          value: 5
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Pods
          value: 2
          periodSeconds: 120
```

## Resource Requirements per Worker

| Resource | Minimum | Recommended | Notes |
|---|---|---|---|
| CPU | 2 cores | 4 cores | Audio processing is CPU-bound |
| Memory | 2 GB | 4 GB | VAD model + connection buffers |
| Network | 10 Mbps | 50 Mbps | ~100kbps per concurrent call |
| Disk | 1 GB | 2 GB | Logs, temporary audio buffers |

## Scaling Bottlenecks & Mitigations

| Bottleneck | Trigger | Mitigation |
|---|---|---|
| LiveKit Server CPU | >200 rooms | Multi-node cluster or LiveKit Cloud |
| LLM API rate limits | >500 concurrent turns/min | Multiple API keys, self-hosted fallback |
| STT connections | >100 concurrent streams | Enterprise tier, secondary provider |
| TTS connections | >100 concurrent streams | Enterprise tier, audio caching for common phrases |
| Redis memory | >10k call records | TTL-based eviction, separate instance for state vs cache |
| ChromaDB queries | >1000 QPS | Qdrant cluster, Redis cache for frequent queries |
